# Product Agent

## Описание

Product Agent - это мета-агент для Cursor, который помогает создавать продуктовую документацию для CRM системы. Он анализирует информацию об организации, определяет MVP scope, создает user stories и описывает бизнес-процессы.

## Назначение

Product Agent используется для:
- Анализа требований к продукту на основе информации об организации
- Определения что входит в MVP (must have, should have, could have)
- Создания user stories для всех ролей системы
- Описания бизнес-процессов работы с клиентами, партнерами и данными
- Формулирования функциональных и нефункциональных требований

## Использование

Когда нужно создать или обновить продуктовую документацию, используй инструкции из:
- `docs/agents/meta-agents/product/instructions.md` - полные инструкции для AI
- `docs/agent-templates/product/` - шаблоны для создания документов

## Входные данные

Product Agent использует следующие входные данные:

1. **Информация об организации**: `docs/product/organization-info.md`
   - Суть проекта и позиционирование
   - Форматы туров и требования
   - Коммуникационная модель
   - Операционные процессы
   - Технические детали для MVP

2. **План проекта**: `PROJECT_PLAN.md`
   - Технологический стек
   - Этапы разработки
   - Общие требования

3. **Текущий MVP scope**: `docs/product/mvp-scope.md` (если существует)

4. **Контекст проекта**: `CONTEXT.md`
   - Архитектура (кратко)
   - Ключевые ограничения
   - Текущий статус

## Выходные артефакты

Product Agent создает следующие артефакты:

1. **Requirements (Требования к продукту)**
   - **Путь**: `docs/product/requirements.md`
   - **Описание**: Функциональные и нефункциональные требования к CRM системе
   - **Шаблон**: `docs/agent-templates/product/requirements-template.md`

2. **User Stories (Пользовательские истории)**
   - **Путь**: `docs/product/user-stories.md`
   - **Описание**: User stories для всех ролей (менеджер, AI-агент, клиент, админ)
   - **Шаблон**: `docs/agent-templates/product/user-stories-template.md`

3. **Business Processes (Бизнес-процессы)**
   - **Путь**: `docs/product/business-processes.md`
   - **Описание**: Описание бизнес-процессов работы с клиентами, партнерами, данными
   - **Шаблон**: `docs/agent-templates/product/business-processes-template.md`

4. **MVP Scope (Обновление)**
   - **Путь**: `docs/product/mvp-scope.md`
   - **Описание**: Обновление MVP scope на основе детального анализа
   - **Примечание**: Если файл существует, обновляется. Если нет - создается новый.

## Примеры использования

### Пример 1: Создание требований для MVP

**Запрос пользователя**:
```
Создай требования к продукту на основе информации о Soul KG
```

**Процесс**:
1. Product Agent читает `docs/product/organization-info.md`
2. Выделяет ключевые требования (общение через WhatsApp, учет клиентов, AI-агенты)
3. Использует шаблон `docs/agent-templates/product/requirements-template.md`
4. Создает `docs/product/requirements.md` с детальными требованиями
5. Обновляет индексы

**Результат**:
- Создан `docs/product/requirements.md` с функциональными и нефункциональными требованиями

### Пример 2: Создание user stories для менеджера

**Запрос пользователя**:
```
Создай user stories для менеджера продаж
```

**Процесс**:
1. Product Agent анализирует процессы из `docs/product/organization-info.md` (раздел 11)
2. Определяет действия менеджера (просмотр клиентов, вмешательство в диалог, передача партнеру)
3. Создает user stories в формате "Как менеджер, я хочу..., чтобы..."
4. Добавляет критерии приемки для каждой story
5. Сохраняет в `docs/product/user-stories.md`

**Результат**:
- Обновлен `docs/product/user-stories.md` с user stories для менеджера

### Пример 3: Описание бизнес-процесса работы с лидом

**Запрос пользователя**:
```
Опиши бизнес-процесс работы с новым лидом
```

**Процесс**:
1. Product Agent использует информацию из `docs/product/organization-info.md` (раздел 11 - этапы 1-6)
2. Определяет участников процесса (клиент, AI-агент, менеджер)
3. Описывает шаги процесса от получения сообщения до квалификации
4. Указывает триггеры, результаты и исключения
5. Сохраняет в `docs/product/business-processes.md`

**Результат**:
- Обновлен `docs/product/business-processes.md` с описанием процесса работы с лидом

## Рабочее пространство

Product Agent имеет следующую структуру рабочего пространства:

```
docs/agents/meta-agents/product/
├── instructions.md          # Инструкции для AI
├── documentation.md         # Документация агента (этот файл)
├── workspace/
│   ├── drafts/             # Черновики документов
│   ├── cache/               # Кэш промежуточных результатов
│   └── temp/                # Временные файлы (не коммитятся)
├── archive/                 # Архивы старых версий
└── logs/                    # Логи работы (опционально)
```

## Связь с другими агентами

Product Agent работает последовательно с другими мета-агентами:

1. **Product Agent** → создает продуктовую документацию
2. **Architecture Agent** → использует требования и user stories для создания архитектуры
3. **Development Agent** → использует архитектуру и требования для создания кода

## Ограничения

- Product Agent **не создает техническую архитектуру** - это задача Architecture Agent
- Product Agent **не создает код** - это задача Development Agent
- Product Agent **фокусируется на продукте** - что нужно пользователям, а не как это реализовано
- Product Agent **учитывает MVP** - не добавляет функции, которые можно отложить

## Вопросы для проработки

При использовании Product Agent проработай вопросы из `docs/product/organization-info.md` (раздел 15):

1. **Архитектура диалога с клиентом**: Один AI-агент или несколько специализированных агентов?
2. **Механизм уведомлений менеджера**: Когда, как, что показывать?
3. **Переключение AI ↔ Человек**: Как менеджер "берет" диалог, как AI продолжает?
4. **Структура продуктов**: Как связаны "наши продукты" и "продукты партнеров"?
5. **Импорт данных из WhatsApp**: Как извлечь и структурировать данные?
